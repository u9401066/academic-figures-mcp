# Decision Log

| Date | Decision | Rationale |
| ---- | -------- | --------- |
| 2026-04-10 | Adopt a dual-deliverable architecture: a local MCP server for agent orchestration and a VS Code extension for Copilot/human workflows, with a structured style preset registry as a core shared layer. | The current repo already has prompt and standards knowledge, but it lacks a reusable data layer for style conversion and a user-facing shell. Making presets a shared structured layer lets both the MCP server and the extension resolve the same journal/style/domain defaults and keeps style transfer explainable and extensible. |
| 2026-04-10 | VS Code extension 以設定切換 google/openrouter provider，並使用 SecretStorage 注入對應 API key 與 OpenRouter headers 到 MCP 啟動環境。 | 這讓本機工作區不必保存明文金鑰，同時讓 OpenRouter 模型 `google/gemini-3.1-flash-image-preview` 與直接 Google Gemini 模式共用同一套 VSX 啟動流程。 |
| 2026-04-10 | 專案與 VS Code extension 一律採用 Apache License 2.0，並在 repo root 與 extension 子專案各自放置一份 LICENSE 檔。 | root LICENSE 可覆蓋整體專案授權，extension-local LICENSE 可讓 vsce 在子專案目錄內打包時正確辨識授權，避免使用 skip-license 繞過缺少授權檔的問題。 |
| 2026-04-10 | The VS Code extension now stores Ollama endpoint/model settings but does not expose Ollama as a working image provider for the Academic Figures runtime. | The Python MCP server currently supports runtime image generation only through Google Gemini and OpenRouter. Storing Ollama settings in the extension is useful for future expansion and external workflows, but presenting it as an active image provider today would be misleading. |
| 2026-04-10 | Academic Figures MCP should be positioned primarily as a complete agent harness for publication-ready figure generation workflows, not merely as a wrapper around an image-generation API. | The core differentiator is the end-to-end workflow: PMID ingestion, figure-type planning, prompt orchestration, quality evaluation, MCP exposure, and VSX packaging so non-engineers can use it quickly. API connectivity is an implementation detail, not the main product story. |
| 2026-04-10 | Do not expand the local PubMed client into a full academic-search middleware; keep search/discovery as an external or sidecar capability and keep this repo focused on figure workflows. | `pubmed-search-mcp` already covers unified literature search, PICO, citation exploration, full-text, and figure extraction. Rebuilding those capabilities here would blur product boundaries and dilute the figure-generation harness. |
| 2026-04-10 | Treat this repository as greenfield and remove legacy compatibility shims instead of preserving old import paths. | No code in the repo depends on the top-level wrapper modules, and the canonical runtime already lives under `src.presentation` and `src.infrastructure`. Keeping dead compatibility layers would only create drift and confuse future contributors. |
| 2026-04-10 | Treat regenerated *.egg-info metadata as build noise and suppress it with workspace settings instead of repeatedly deleting it. | uv/setuptools workflows can recreate egg-info during normal validation. Hiding *.egg-info from Explorer/search and excluding it from markdownlint removes editor noise without changing packaging behavior. |
| 2026-04-11 | Align workspace `.vscode/mcp.json` with the official VS Code MCP schema by using `type: "stdio"` and `envFile`, while keeping PowerShell only for workspace directory switching. | The current VS Code documentation explicitly supports `envFile` for stdio servers and recommends avoiding hardcoded secrets in `mcp.json`. This keeps the shared config closer to the platform model and avoids custom secret-handling logic in the workspace config itself. |
| 2026-04-11 | Keep [scripts/start_afm_local.ps1](scripts/start_afm_local.ps1) as the manual bootstrap path for local `afm-server` and `afm-run` flows. | The wrapper is still useful outside `mcp.json` because local shells do not automatically load the repo-root `env` file. It provides one repeatable entrypoint for manual testing, smoke runs, and direct figure generation. |
| 2026-04-11 | Public bitmap creation should converge on one generic generation tool, while PMID intake, planning, evaluation, presets, and workflow controls remain harness layers around it. | The product moat is the workflow harness rather than multiple competing render entrypoints. A single generic generation contract can later support academic figures, icons, and other visual assets, while domain-specific tools keep shaping, validating, and sequencing requests upstream. |
| 2026-04-11 | Implement `generate_figure` as a dual-input contract that already accepts generic `planned_payload`, while keeping direct PMID generation only as a backward-compatible bridge. | This lets the repo start using one public renderer immediately without breaking existing CLI, tests, or extension flows. New asset types such as repo icons can now use the same generation tool through a payload-file path. |
| 2026-04-10 | Use a YAML-backed journal registry as the source of truth for figure-house-style constraints and inject them from the MCP layer. | This removes the need for agents to memorize journal rules, keeps prompt modifications deterministic, and allows planners/generators to switch journal standards through a data file instead of hard-coded prompt text. |
| 2026-04-11 | Maintain root ROADMAP.md and CHANGELOG.md as first-class project documents in addition to Memory Bank notes. | Product direction and shipped changes should be visible to contributors and users directly in the repository, not only in internal Memory Bank files. |
| 2026-04-11 | Standardize repo-level MCP startup on a shell-neutral `uv run --project ... python -m ...` pattern and provide a Python local launcher for manual use. | The static workspace config must not depend on PowerShell if the repo is expected to work unchanged on Windows, macOS, and Linux. Keeping the launch path in `uv` and Python preserves one portable startup contract while allowing the PowerShell helper to remain as a Windows convenience wrapper. |
| 2026-04-11 | Keep direct PMID input only as a compatibility bridge and force `generate_figure` itself to render from a planning payload. | Planning and generation should no longer diverge into separate code paths. Keeping one render contract reduces drift, preserves reusable `planned_payload` artifacts, and makes explicit planning the canonical workflow without breaking existing callers immediately. |
| 2026-04-14 | Composite figure assembly must follow the same Presentation -> Application -> Domain interface -> Infrastructure path as the other MCP tools. | The previous presentation-level implementation duplicated orchestration logic and bypassed the DDD boundary. A dedicated CompositeFigureUseCase keeps tool handlers thin, makes the route testable, and aligns composition with the existing DI/governance model. |
| 2026-04-14 | Publish Python releases to PyPI through GitHub Actions OIDC Trusted Publisher, and publish the VS Code extension from the same tag-triggered workflow. | This removes long-lived PyPI API tokens, keeps release execution tied to reviewed Git tags, and gives Python package and VS Code Marketplace artifacts one consistent release contract. |
| 2026-04-14 | Determine output image media type from the actual returned bytes, not from the filename or a hard-coded `.png` assumption. | Gemini/OpenRouter can return JPEG bytes even when the surrounding workflow expects a PNG path. Detecting media type from signatures at the adapter boundary keeps file extensions, edit outputs, smoke tests, and committed assets consistent with the real payload. |
| 2026-04-14 | Keep MetadataFetcher as the narrow application boundary, and make offline/local corpora a separate file-backed infrastructure adapter selected by config. | This delivers the first real pluggable metadata source without coupling the figure runtime to MCP transport, PyPI search libraries, or richer research payloads. It also improves smoke tests and demos by allowing deterministic local paper inputs. |
| 2026-04-14 | AFM requires a patch release after VSX smoke testing because the published extension manifest was missing mcpServerDefinitionProviders, which breaks activate() in the extension host. | Backend smoke passed, but extension-host smoke exposed a real runtime failure in the published VSX package. The manifest fix is minimal, validated locally, and user-facing, so it should ship as a patch release rather than wait for a larger feature batch. |
| 2026-04-14 | AFM should document uvx-based package mode as the primary cross-platform install path, following the Zotero Keeper pattern. | `uvx --from academic-figures-mcp afm-server` is shell-neutral and avoids repo-path, quoting, and platform-specific launcher issues. Local checkout mode remains for development, but package mode is the safest default for macOS/Linux/Windows users. |
| 2026-04-14 | The package smoke workflow must force UTF-8 output for the uvx child process. | Windows GitHub Actions runners default to cp1252 console encoding, and AFM planning output includes Unicode arrows and CJK-safe text guidance. Forcing UTF-8 keeps the cross-platform smoke deterministic without changing user-facing package behavior. |
